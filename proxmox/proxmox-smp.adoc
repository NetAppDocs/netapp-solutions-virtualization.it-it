= Configurare e utilizzare Storage Manager per Proxmox
:allow-uri-read: 
:icons: font
:imagesdir: ../media/
:linkattrs: 


[role="lead"]
Storage Manager for Proxmox (SMP) offre provisioning dello storage e gestione del ciclo di vita completi per le macchine virtuali negli ambienti Proxmox che utilizzano sistemi storage NetApp ONTAP. SMP consente agli amministratori di gestire lo storage direttamente, semplificando le operazioni grazie a funzionalità automatizzate di provisioning, backup, ripristino e clonazione.



== Scopri Storage Manager per Proxmox

Storage Manager for Proxmox (SMP) offre una gestione end-to-end dello storage per le macchine virtuali all'interno degli ambienti Proxmox che utilizzano NetApp storage systems. Consentendo agli amministratori di gestire direttamente lo storage, SMP semplifica sia le attività di gestione dello storage che quelle di gestione dei dati negli ambienti Proxmox.



=== Caratteristiche principali

* *Gestione completa dello storage*: SMP offre una gestione completa del ciclo di vita delle macchine virtuali, dal provisioning dello storage al backup e al ripristino.
* *Gestione diretta dello storage*: gli amministratori di Proxmox possono gestire lo storage direttamente, semplificando le operazioni e riducendo la complessità.
* *Programma di installazione autonomo*: SMP viene fornito come programma di installazione autonomo che si connette senza problemi ai nodi Proxmox tramite API.




=== Capacità

Ogni componente di SMP è progettato per migliorare l'efficienza della gestione dello storage:

* *Provisioning dello storage*: Assegna e gestisci facilmente risorse di storage per gli host Proxmox e le macchine virtuali associate.
+
** *Provisioning dello storage NFS*: Configura e imposta lo storage NFS di ONTAP da utilizzare con il tuo ambiente Proxmox.
** *iSCSI provisioning dello storage*: Implementare LVM thick con multipath per una gestione dello storage iSCSI robusta ed efficiente.


* *Backup e ripristino*: Esegui backup affidabili e ripristini rapidi per garantire l'integrità e la disponibilità dei dati.
+
** *Backup coerenti con gli arresti anomali e con i file*: Garantisci l'integrità dei dati configurando meccanismi di backup sia coerenti con gli arresti anomali che con i file.
** *Operazioni di ripristino delle macchine virtuali*: Esegui i processi di ripristino delle macchine virtuali per recuperare i dati e mantenere la disponibilità del sistema.


* *Efficienza dello storage*: ONTAP offre una suite completa di tecnologie per l'efficienza dello storage, progettate per ridurre al minimo lo storage necessario per i dati. Queste tecnologie non solo riducono il consumo di storage, ma migliorano anche le prestazioni del sistema, offrendo:
+
** *Provisioning rapido delle macchine virtuali*: Semplifica il provisioning delle macchine virtuali per una distribuzione più rapida.
** *Copie dei dataset più rapide*: velocizza la duplicazione dei dataset per una gestione efficiente dei dati.






=== Prerequisiti

Assicurati di aver completato i seguenti passaggi prima di procedere:

* Cluster Proxmox Virtual Environment (PVE) 9.x o versione successiva funzionante.
* Storage ONTAP All Flash FAS (AFF) con protocollo NFS e iSCSI abilitati a livello SVM. Per istruzioni, vedere link:proxmox-ontap-nfs.html["Configura i datastore NFS con ONTAP"] e link:proxmox-ontap-iscsi.html["Configurare Proxmox VE 9.x per iSCSI con storage ONTAP"].
+
[NOTE]
====
Per la configurazione di una LUN iSCSI, è necessario abilitare quanto segue:

*Installare multipath e iSCSI su tutti gli host Proxmox:*

[source, shell]
----
apt update
apt install open-iscsi multipath-tools -y
systemctl enable --now iscsid multipathd
----
*Configura iSCSI initiator e connettiti tramite iSCSI (sugli host Proxmox):*

Crea o aggiorna `/etc/multipath.conf`:

[source, shell]
----
touch /etc/multipath.conf
----
Oppure usa:

[source, shell]
----
cat << 'EOF' > /etc/multipath.conf
defaults {
   user_friendly_names yes
   find_multipaths yes
}
blacklist {
   devnode "^sd[a]$"
}
EOF
----
====
* Gli host Proxmox e lo storage ONTAP sono raggiungibili in rete dall'host SMP.

Il primo passo consiste nell'individuare e aggiungere l'endpoint di storage ONTAP e poi i nodi Proxmox di destinazione all'SMP.

.Prima di iniziare

== Provisioning dello storage

Il provisioning di un volume come pool di storage, comunemente denominato datastore, crea un container logico per le macchine virtuali e i relativi dischi virtuali (QCOW2 o RAW). SMP consente di effettuare il provisioning di un datastore e di collegarlo a un singolo host o a tutti gli host all'interno di un cluster.

=== Opzioni di configurazione del volume

Quando SMP esegue il provisioning di nuovi ONTAP FlexVol volumi e LUN, vengono applicati automaticamente diversi valori di configurazione predefiniti:

[cols="2*"]
|===
| Impostazione ONTAP | Valore predefinito 


| riserva di Snapshot (-percent-snapshot-space) | 0% 


| Riserva frazionaria (-fractional-reserve) | 0% 


| Aggiornamento dell'ora di accesso (-atime-update) | Disattivato 


| Previsione minima (-min-readahead) | Disattivato 


| Snapshot programmati | Nessuno 


| efficienza di storage | Abilitato 


| Garanzia di volume | Nessuno (thin provisioned) 


| dimensionamento automatico | aumenta_riduci 


| Riserva di spazio LUN | Disattivato 


| allocazione dello spazio LUN | Abilitato 
|===


=== Comportamento di NConnect

Quando si utilizza lo storage NFS con Proxmox VE, è importante tenere presente il comportamento specifico del client NFS (come utilizzato da Debian, il sistema operativo di base di Proxmox VE). Il kernel Linux potrebbe riutilizzare i parametri di montaggio dalla prima volta che un'esportazione NFS viene montata su un nodo. Se la stessa esportazione viene montata nuovamente con opzioni di montaggio diverse, queste nuove opzioni potrebbero non essere applicate.

Come best practice, crea e monta esportazioni separate e utilizza indirizzi IP del server NFS diversi, come LIF di dati distinti. SMP rileva automaticamente le connessioni esistenti insieme ai relativi valori nconnect e applica il valore nconnect appropriato a qualsiasi nuova connessione.



== Proteggere le macchine virtuali

Quando le macchine virtuali vengono eseguite su datastore NFS supportati da ONTAP, l'approccio di protezione più efficiente consiste nel lasciare che Proxmox stabilisca la coerenza del filesystem, mentre ONTAP acquisisce il punto di backup effettivo a livello di storage. Le Snapshot ONTAP forniscono copie point-in-time rapide ed efficienti in termini di spazio del volume del datastore e possono essere utilizzate successivamente per ripristinare interi datastore, singoli file o contenuti di VM.

SMP offre un semplice orchestratore per la protezione dei dati coerente con i crash e con le macchine virtuali e la gestione dei cloni. Questo software semplifica i cicli di vita di backup, ripristino e clonazione attraverso la creazione di policy di backup applicabili a più processi. Queste policy possono definire pianificazione, conservazione, replica e altre funzionalità. Consentono la selezione facoltativa di snapshot coerenti con i file; questo approccio sfrutta la capacità di QEMU di sospendere le operazioni di I/O prima di acquisire una Snapshot ONTAP.

SMP consente di proteggere un intero datastore, dove un volume NFS ONTAP contiene più macchine virtuali, gruppi di macchine virtuali che possono estendersi su più datastore, oppure a livello di singola macchina virtuale.

=== Dettagli del flusso di lavoro di backup

Il flusso di lavoro di backup high-level comprende:

. *Definisci l'ambito di protezione*: Seleziona se la destinazione della protezione è un datastore (volume ONTAP), un gruppo di risorse (più volumi o VM) o una singola VM.
. *Conferma il metodo di coerenza*: Scegli come viene creata la coerenza prima dello snapshot ONTAP:
+
** Blocco/sblocco dell'agente guest QEMU per la quiescenza del filesystem all'interno della macchina virtuale e l'integrazione con Windows VSS.


. *Esegui il dump dei file di configurazione della macchina virtuale*.
. *Crea una Snapshot ONTAP*: acquisisci una Snapshot ONTAP del volume del datastore mentre le VM si trovano in uno stato coerente. Questa diventa la copia di backup conservata.
. *Rimuovi gli snapshot temporanei di Proxmox*: Elimina gli snapshot di Proxmox dopo il completamento dello snapshot ONTAP per evitare la crescita dei file delta.
. *Applica le policy di conservazione e replica (utilizzando le policy)*: applica la policy di conservazione ONTAP configurata e replica il set di Snapshot se la replica per il disaster recovery è abilitata.


Questo approccio riduce al minimo lo spazio di storage e la larghezza di banda memorizzando solo le informazioni modificate. Utilizza le copie Snapshot per creare copie rapide della macchina virtuale o del datastore senza influire sulle prestazioni e poi inviale a un sistema secondario utilizzando la tecnologia NetApp SnapMirror per la protezione dei dati a lungo termine e fuori sede.

=== Dettagli del flusso di lavoro di ripristino

Il flusso di lavoro di ripristino varia a seconda del tipo di storage sottostante e prevede i seguenti passaggi chiave:

. *Definisci l'ambito del ripristino*: Seleziona se la destinazione della protezione è un datastore (volume ONTAP), un gruppo di risorse (più volumi o VM) o una singola VM.
. *Eseguire l'operazione di ripristino*: Il processo di ripristino della macchina virtuale varia a seconda del protocollo storage:
+
** *Ripristini basati su NFS*: Quando si ripristina un volume o un datastore su NFS, SMP avvia un'operazione di ripristino dello snapshot del volume sul volume di destinazione. Per le VM ospitate su volumi NFS, SMP convalida i parametri di ripristino, inclusi l'ID della VM, il nome dello snapshot, gli ID del sito e il tipo di storage. Se la VM è in esecuzione, SMP la arresta verificandone lo stato ed emettendo un comando di arresto. Il sistema ripristina quindi i dischi della VM dallo snapshot utilizzando un'operazione di ripristino a livello di file.
** *Ripristini basati su iSCSI*: Per i datastore o i LUN iSCSI, SMP ripristina il LUN dallo snapshot, verifica il dispositivo ripristinato, esegue una nuova scansione iSCSI sui nodi Proxmox e garantisce l'accessibilità del dispositivo. Per le VM residenti su storage iSCSI, SMP clona i LUN dello snapshot, mappa i LUN clonati sull'host Proxmox ed esegue il rilevamento iSCSI per importarli. Lo storage ripristinato viene quindi aggiunto e le configurazioni delle VM vengono aggiornate. I dischi delle VM vengono spostati nuovamente nello storage originale e l'avanzamento viene monitorato per ogni VM. Il processo si conclude con attività di pulizia, tra cui la rimozione dello storage temporaneo e l'eliminazione dei LUN clonati.

=== Dettagli del flusso di lavoro di clonazione

SMP automatizza la clonazione rapida delle macchine virtuali delegando le operazioni più complesse a ONTAP, che esegue cloni lato storage anziché copie di file lato host. A livello interno, SMP sfrutta la semantica FlexClone di ONTAP (cloni basati su snapshot, scrivibili ed efficienti in termini di spazio) in modo che i cloni vengano creati rapidamente e consumino inizialmente poca capacità; inoltre, evita, ove possibile, operazioni di lettura/scrittura dei dati tramite l'host Proxmox.

Questo approccio accelera il provisioning delle macchine virtuali spostando il lavoro di clonazione in ONTAP, fornendo cloni veloci, leggeri ed efficienti in termini di spazio, adatti a Dev/Test, laboratori e cicli di aggiornamento rapidi.

Per impostazione predefinita, la clonazione viene eseguita sul file system attivo. Il flusso di lavoro di clonazione consente anche la selezione di uno Snapshot, nel qual caso i cloni dei file vengono eseguiti utilizzando un nuovo volume creato dallo Snapshot selezionato.
